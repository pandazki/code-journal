# Roadmap

Where code-journal is heading, and the work to get there.

## Direction

code-journal is a **local, project-scoped tool** that turns your AI coding-agent
sessions into a **visual journal** of your work.

- You run `code-journal` inside a project; it starts a local web server and
  opens a browser onto that project's journal.
- **Local-first, zero-config.** It reads your local agent sessions
  (`~/.claude/projects/`, Codex, Cowork) directly — no upload, no account, no
  setup needed to get value.
- The **journal** is the centerpiece: not a list of worklog entries, but a
  visual, time-organized narrative of your work.
- Uploading sessions to S3, pushing reports to a remote server — **optional
  extensions**, configured in the GUI, off by default.

The current Ink TUI is superseded by the web GUI: a journal needs real visuals
(timelines, heatmaps) a terminal can't do well. The discovery and
transcript-parsing logic in `packages/core` carries over unchanged; so does the
transcript-viewer SPA in `server/public/`.

## Decisions (resolved 2026-05-22)

- **Journal unit = the day**, with a git-contribution-tile feel — but the day is
  one of two co-equal lenses, not the only one.
- **Two lenses, both shipped:**
  - **Day lens** — a day's work, tile / heatmap-driven.
  - **Project lens** — how a project gets built bit by bit over time (the old
    "worklog" perspective, renamed away from "worklog").
- **Scope: project-scoped *and* a cross-project overview.** Run inside a
  project for that project's journal; an all-projects overview sits above it.
- **The day's narrative is generated by the host coding agent.** Reuse the old
  project's approach: shell out to the user's already-installed coding agent
  (`claude -p`, …) to write the narrative — zero marginal cost, no API key, no
  account. Pure-metadata cards work with no agent at all; the narrative is the
  enrichment layered on top.
- **The Claude Code plugin folds into the journal.** Its worklog / report
  generation becomes the journal's narrative engine — one product line, not two.

## A — Local web GUI

- [x] `packages/app` — local server (`node:http`, zero-dep) + the SPA, opens the browser
- [x] Local data source: single-pass discovery from local sessions (`core/journal-fs`), no upload
- [x] Editorial SPA graduated from the prototype; degrades to metadata-only when no narrative
- [x] Per-day session splitting — a resumed session counts toward every day it spans
- [x] `code-journal` / `cj` bin — build + serve, or `code-journal narrate`
- [ ] Move configuration (S3, remote server) into the GUI; optional, never a gate
- [ ] Retire / shrink `packages/tui` (the Ink board) — keep its logic in `core`
- [ ] Keep a headless `code-journal sync` for the cron path

## B — The journal (the centerpiece)

- [x] core: journal model — group discovered sessions into (project, day) entries
- [x] core: metadata phases — segment a project's life into chapters at activity gaps
- [x] **Overview** — cross-project home: ink-density heatmap + recent stream + project index
- [x] **Project view** — a project's arc as a book: opening, heatmap, collapsible chapters
- [x] **Day view** — the day's entry: marginalia, headline, sessions with file/command detail
- [x] core: merge git commits into the day entry (author-filtered, bucketed by date)
- [x] Narrative engine — host coding agent (`claude -p`) writes day titles + day / project recaps, cached to disk
- [x] drill from a session into its raw transcript — line-numbered entry viewer
- [ ] Fold the plugin's worklog generation into the narrative engine

## C — Onboarding & adoption

- [x] First run = zero-config, straight to the journal — `code-journal` just works, no setup
- [ ] Prebuilt release binaries + a one-line install; tag a release, wire `release.yml`
- [ ] (idea) a "local archive directory" backup target, so backup needs no cloud

## D — Extensions (optional, user-configurable)

- [x] S3-compatible session upload
- [x] Push worklog / reports to a remote server (`server/`)
- [x] Scheduled upload via cron
- [ ] Define clean extension points so users can add their own targets

## E — Observation lens (the long-form direction, 2026-05-27+)

A separate product line growing alongside the journal: take the same raw
sessions and run them through observation lenses that surface where the
user (not the agent) injected direction. **Shipping as of v0.2.0** — the
canonical, code-matching documentation is
[`docs/observation-lens.md`](observation-lens.md) (中文:
[`observation-lens.zh.md`](observation-lens.zh.md)). The exploratory record
(phases, hypotheses, re-validation) lives under `experiments/observation-lens-*/`
and `docs/plans/` and is superseded by that guide for day-to-day use.

- [x] Phase 1 · cross-project comparison (3 projects × 2 lenses, isolated subagents) — `experiments/observation-lens-v1/report.md`
- [x] Phase 2 · seven-experiment battery (lens variance, empty-state, measurement utility, cross-agent, fate proxy, predictability proxy, third-party reader proxy) — `experiments/observation-lens-v2/report.md`
- [x] MVP-II · production three-layer architecture (Detection / Signal Store / Audit) — `sync` / `compose` / `status`
- [x] v3 re-validation — adversarial grounding gate, `assented` stance (de-contaminate `engaged`), third lens `user-initiated-pivot`, generalization on unseen projects — `experiments/observation-lens-v3-revalidation/`
- [ ] MVP-III — **not yet built/validated:** fate tracking (manual today; auto-detection + `fate add` CLI), large-session chunked digest, cross-machine sync, real-human reader tests, § 9 predictability dashboard
- [ ] Tune `user-initiated-pivot` soft edges (session openers, question-form concern-surfacing) — currently experimental
